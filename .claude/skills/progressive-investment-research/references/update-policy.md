# 更新策略

更新目标是维护判断复利，不是制造文档负担。

## 更新类型

`no model change`：回答没有改变模型。

`minor update`：低风险补充，例如新增来源指针、澄清定义、补充开放问题、关闭一个验证条件。

`major update`：核心判断、模块边界、比较结论、重要假设或 Current Model 发生新增、加强、削弱或矛盾。

`requires restructure`：当前范围、model map 或模块边界已经不适配，需要 scope refactoring。

## 什么时候写文件

必须写：

- 冷启动后的第一版 Current Model。
- major update。
- promote 后。
- scope split / merge / archive 后。

可以只口头输出：

- 简单问答。
- no model change。
- 一次临时搜索尚未改变判断。

## 需要用户确认

- 改变顶层判断。
- 重写 `model-map.md`。
- promote 到 Obsidian 主知识库。
- scope split / merge / archive。
- 把未验证假设提升为稳定结论。

## 冲突处理

遇到冲突先标记，不自动调和：

- 事实冲突。
- 定义冲突。
- 解释冲突。
- 预测冲突。
- 框架冲突。

至少记录：双方说法、来源、可能原因、临时判断、需要什么证据解决。

冲突可以进入 Current Model，但必须保持可见。

## 审计触发

以下情况应先审计：

- promote 前。
- major update 前，尤其会成为后续依据的判断。
- 来源主要来自派生材料或 agent 推断。
- 判断依赖少数敏感数字、模型输出或关键假设。
- 数字看起来像非公开信息、派生估算或难以公开搜索复现。
- 主 agent 发现自己在解释冲突而不是解决冲突。
- 用户要求检查逻辑、反方、审计或证据链。

审计只输出 findings。主 agent 负责采纳、拒绝或列出冲突。

## Grill 触发

`grill me` 是研究对话，不是审计。它用于了解用户对行业的先验、经验判断和隐含假设，并把这些内容转化为可追踪的模型输入。

触发时机：

- 用户明确要求 grill / 问我 / 讨论推进。
- 已读材料后，开放问题依赖用户内部事实、行业经验、投资判断或风险偏好。
- 某个关键数字看起来像非公开信息，而最需要问清的是用户如何获得、理解或使用它。
- 当前模型有多条合理分支，需要先理解用户更相信哪条机制以及为什么。

不触发：

- 可以直接从本地材料、公开搜索或脚本得到答案。
- 当前问题还没有定位到具体分析轴，问了也只会泛泛而谈。
- 用户要求的是反方、证据链或一致性检查。

每轮 grill 之后，主 agent 应把用户回答归类为 `user-prior`、`judgment`、`assumption`、`source-lead` 或 `open_question`。如果回答改变核心判断，进入 major update；如果回答暴露证据链风险，再另行触发审计。

## Update Log

`update-log.md` 只记录模型变化，不记录编辑过程。推荐字段：

```markdown
| 日期 | 类型 | 变化 | 影响 | 来源 |
| --- | --- | --- | --- | --- |
```

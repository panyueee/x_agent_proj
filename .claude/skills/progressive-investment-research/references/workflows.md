# 工作流

流程是默认路径，不是仪式。只做能推进当前模型的动作。

## 0. 定位

任何研究动作前，先判断用户真正要推进什么：

1. 研究对象：公司、细分行业、产业链、技术路线、政策、市场或主题。
2. 当前目的：冷启动、材料吸收、回答问题、更新模型、处理冲突、status、promote、scope。
3. 输出用途：口头状态、dossier 更新、Obsidian 入库、会议准备、进一步搜索。
4. 是否已有材料足够推进。
5. 是否存在 `context.md`，以及它是否仍能正确说明接续入口。

能安全假设时先推进，并说明假设。只有内部事实、偏好、目标冲突、promote 范围等无法自行判断的问题才问用户。

## 冷启动

用户通常会给足量启动材料。冷启动目标是处理第一批材料，建立第一版 Current Model，不是空白规划。

1. 创建或定位 dossier。
2. 建立最小 `context.md`：研究对象、默认 skill、接续入口、写入触发和本地约定。
3. 阅读用户提供的启动材料。
4. 区分事实、判断、假设、开放问题和来源类型。
5. 起草最小 `model-map.md`：边界、核心问题、分析轴、首批模块、开放问题。
6. 起草 `current-synthesis.md`：当前一句话判断、已知、不确定、冲突、下一步。
7. 写入 `open-questions.md` 中会影响判断的问题。
8. 如果启动材料包含关键数字，记录数字锚点：数值、单位、时间点、来源、来源类型、复核状态、用途。
9. 如果启动材料已经包含可复用公式、参数表、倒推关系或敏感性，建立最小建模入口或模型 backlog。
10. 只创建必要模块；不要为了完整性铺目录。

冷启动后可以进入逐点填充，但不要在没有 map 的情况下直接 tracer bullet。

## 日常研究

1. 如果存在 `context.md`，先读它，恢复工作区入口、默认 skill 和写入规则。
2. 读 `current-synthesis.md`。
3. 读 `model-map.md`，定位问题落在哪个分析轴或模块。
4. 只读取相关模块和开放问题。
5. 判断是否需要搜索、并行 subagent、计算、比较或冲突处理。
6. 综合材料，区分事实、判断、假设、未知。
7. 判断模型是否变化：
   - 没有变化：只回答。
   - 小变化：建议或写入轻量补丁。
   - 大变化：输出模型更新提案，必要时审计。
8. 阶段性结束时输出 Status Brief。

### Context 写入规则

`context.md` 只在入口协议变化时更新，不在每次讨论时更新。

应写入或更新：

- 新建 dossier / workspace。
- 用户明确说“写入 context”“更新 context”“以后默认这样接续”。
- 研究对象、边界、默认 skill、入口文件、验证方式或本地路径发生变化。
- 多次接续都需要重复解释同一背景，说明恢复成本过高。

不应默认写入：

- 用户只是问一个事实、表达直觉、讨论机制或要求口头解释。
- 观点尚未讨论清楚，用户还没有确认要沉淀。
- 内容属于研究结论，应进入 `current-synthesis.md`、模块、模型或开放问题，而不是 context。

如果不确定，先口头回答并说明“这段暂不写入；你确认后我再更新 context”。

## 数字吸收

数字不是装饰。凡是会影响判断的数字，都应从文字材料中被单独捕捉。

记录字段：

- 数值和单位。
- 时间点和统计口径。
- 来源和来源类型。
- 复核状态：`verified-primary` / `needs-primary-check` / `secondary-estimate` / `assumption` / `derived`。
- 用途：事实锚点、模型输入、模型输出、反证指标。

特别注意：

- 私营公司收入、客户数、合同金额、真实成交价、云 credits、产能分配等“非公开感数字”必须标出来源边界。
- `CapEx`、`PPE`、`lease`、`purchase commitment`、`ARR`、`RPO`、`backlog`、`run-rate` 不能混用。
- MAU、DAU、token 调用、下载量、收入、毛利、补贴不是同一层指标，不能互相替代。

如果数字来自用户材料或派生材料但公开搜索不容易复现，不要删掉；保留为线索并标记 `needs-primary-check` 或 `secondary-estimate`。

## 建模

当研究问题依赖公式、参数、倒推、层间传导或敏感性时，进入 modeling。

适合建模的情况：

- 单位经济：单 GPU、单机架、单客户、单 workflow、单 token 或单任务。
- CapEx 回收：现金支出、租赁、采购承诺、RPO/backlog、收入确认。
- 供应链倒推：从 token / revenue / workload 需求倒推 GPU、HBM、CoWoS、基板、电力和机房。
- 瓶颈排序：哪个环节最慢，扩产后瓶颈迁移到哪里。
- 商业化桥接：MAU → 付费率 → ARPU → token 成本 → 毛利。

建模输出：

- 若只是临时推理，可以在回答中给公式和关键敏感性。
- 若会复用或成为后续依据，应写入 `models/`。
- 模型必须写清参数来源、单位、假设、敏感性、局限和支持/挑战的判断。
- 若模型包含多步算术、倒推链或口径转换，必须写 `Calculation Sheet`，用 `row_id` 记录每一行输入、公式、输出、单位、来源状态和解释。

不要追求虚假精确。缺参数时列缺口和最小可用模型。

模型更新顺序：

1. 先定位变化影响的是输入、公式、假设、敏感性还是当前结论。
2. 若有 `Calculation Sheet`，先更新 canonical rows，并保留 `row_id` 的稳定性；废弃行不要静默复用旧编号。
3. 再更新 `当前结果`、`敏感性` 和相关模块中的文字读数。
4. 最后更新 `current-synthesis.md` 或 `update-log.md`。若只是内部行式修正且不影响判断，可只更新模型文件。
5. 如果需要 Excel / CSV / research graph，默认把它当派生视图，不作为新的事实源。

## Grill Me / 研究对话

`grill me` 是对用户先验的结构化访谈，用来推进 shared understanding。它不是审计、反方或来源核验；审计由审计型 subagent 或主 agent 的 evidence check 处理。

启动前提：

- 已读当前 dossier 或冷启动材料，能把问题定位到具体分析轴、模块或开放问题。
- 能区分这个缺口是用户内部先验/判断框架，还是公开事实/来源核验。

触发：

- 用户明确要求 grill / 问我 / 讨论推进 / 把我的想法问出来。
- 开放问题依赖用户的行业经验、投资偏好、风险承受、交易经验或对某个数字的来源记忆。
- 当前模型有多条合理分支，需要先了解用户为什么更相信某条机制。
- 有非公开感数字、派生估算或模型参数，但关键问题是用户当时如何获得、理解或使用这个数字。
- 搜索已经不足以继续收敛，剩下的是判断框架和假设选择。

不触发：

- 能通过本地材料、脚本、公开搜索或 subagent 查证的问题。
- 还没读材料，提不出具体问题。
- 用户要求审计、反方或证据链检查。
- 问题不影响 Current Model、开放问题或后续研究方向。

提问格式：

```markdown
1. 问题：
   为什么问：
   影响模型：
   你可以这样回答：
   如果暂时不知道：
```

默认一次只问 1 个问题。若用户要一轮 grill，最多问 3-5 个互不依赖的问题。不要把互相依赖的问题堆给用户。

回答后的处理：

- 用户内部事实或来源记忆：标为 `source-lead` 或 `needs-primary-check`，后续再核验。
- 用户行业直觉或投资判断：标为 `user-prior` 或 `judgment`。
- 用户不确定但有方向感：标为 `assumption`，进入模型敏感性或开放问题。
- 可公开查证部分：转为搜索或 subagent 任务。

## Status

`status` 不做新研究，只恢复当前模型，让用户知道我们在哪。

输出：

- 当前一句话判断。
- 当前 map 摘要。
- 已知 / 不确定 / 冲突。
- 最近模型变化。
- 最重要的 3-5 个开放问题。
- 未处理材料或风险。

当用户担心“AI 做了什么我不知道”时，优先执行 status。

## 并行搜索

适用于公开资料、竞品、政策、交易案例、价格、论文、数据口径等可独立检索的问题。

规则：

- 每个 subagent 只负责一个明确问题。
- 可使用低成本模型。
- 输出 findings，不编辑 dossier。
- 主 agent 负责来源分层、去重、冲突处理和模型更新。
- 不把搜索结果直接当结论。

## 审计

不只 promote 需要审计。以下情况触发审计：

- promote 前。
- major update 前。
- 核心判断冲突或来源薄弱。
- 用户要求反方、审计、逻辑检查。
- 主 agent 产出将成为后续研究依据。

审计默认使用独立上下文 subagent，只读输出反对点、证据缺口和建议修改。

## Promote

Promote 是把 Research dossier 中少量高价值成果提升到 Obsidian 主知识库，不是把整个 dossier 搬进去。

1. 选择候选成果：稳定判断、概念定义、精炼模块、分析框架或已审计开放问题。
2. 主 agent 整理 promote 草案。
3. 独立 subagent 审计。
4. 主 agent 处理审计冲突，明确剩余不确定性。
5. 写入 Obsidian：挂入 MOC、补 frontmatter、建立语义链接。
6. 使用 Obsidian CLI 验证搜索、回链、属性、标签和出链。
7. 在 `update-log.md` 记录 promote 结果。

允许带未解决问题入库，但必须标注为待验证假设或开放问题。

## Scope Refactoring

研究范围不是由题目大小决定，而是由 Current Model 是否还能让人类把握决定。

触发信号：

- `current-synthesis.md` 不能在 30-60 秒内恢复全局状态。
- 分析轴超过 5-7 个且彼此相对独立。
- 某个模块膨胀成独立研究对象。
- 子主题更新节奏完全不同。
- 同一开放问题反复牵扯多个模块。

输出 keep / split / merge / archive 建议。默认先诊断，不立即搬迁文件。

拆分时最小迁移：新建子 dossier，继承必要判断和开放问题；父 dossier 保留链接和边界说明。

合并时吸收稳定判断，避免双事实源。
